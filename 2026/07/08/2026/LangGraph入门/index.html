LangGraph入门

最近在做一些关于agent skill组织、管理与调用的研究，使用的框架是LangGraph，这篇帖子用来记录我入门该框架的学习过程

Type Annotations：LangGraph中常用的类型注解

dictionary

moive = {"name":"EA","year":2019}

优点：高效地检索数据、flexbile；
缺点：无法确保是正确的数据类型或结构（大项目中导致问题出现）

类型字典

from typing import TypedDict

class Moive(TypedDict):
    name : str
    year : int
movie = Movie(name="EA",year=2019)

优点：确保类型安全性，代码可读性提升；
在LangGraph大量使用类型字典，定义状态；

联合类型Union

from typing import Union
def square(x: Union[int,float]) -> float:
    return x * x
x = 5
x = 3.14
x = "hello" # 不可用于square

Union让一个值可以是更多的类型，较为灵活；
但是它也提供了类型检查，一定程度确保了安全性，例如上面的例子中x是int或float是可以的，但是str则会报错；
在LangGraph中Union也被广泛使用；

可选类型Optional

from typing import Optional

def nice_message(name: Optinal[str]) -> None:
    if name is None:
        print("no input parameters")
    else:
        print(f"Hi there, {name}!")

这里定义的必须是字符串或者空值，但是这个参数是可选的，在没有参数传入的时候Optional这个可选类型的价值就体现出来了。

Any

from typing import Any

def print_value(x: Any)
    print(x)

可以接受任何类型

lambda函数

square = lambda x: x*X
square(10)

编写小型函数的快捷方式，让一切变得高效

nums = [1,2,3,4]
squares = list(map(lambda x: x*x,nums)) # map的作用是对nums中每个值进行给定函数的操作，最终转换回list，相当于对原数组平方，这比for循环高效

Element：LangGraph中的基础元素

State：共享数据结构，存储整个应用程序的当前信息或上下文。node在执行时可以访问或修改这些内容

Nodes：graph中执行特定任务的独立函数或操作，node接受输入通常就是state，处理后给出输出或更新state

Graph：总体结构，描绘不同任务是如何连接和执行的，可视化的方式呈现工作流程，展示了操作的条件、顺序

Edges：节点之间的连接，决定流程，告诉node在当前任务完成后，下一个应该启动哪个node的执行

Conditional Edges：基于当前状态应用的特定条件或逻辑，选择下一个node

Start：起始节点，langgraph中的虚拟入口，标志工作流程从哪里开始（本身不执行任何操作，是专门的起点）

End：终止节点，langgraph中工作流程的结束，到达时表明所有工作已经执行结束

Tool：专门的函数或工具，提供给node使用

ToolNode：特殊的节点，主要作用是运行工具

StateGraph：LangGraph的一个类，用于构建和编译图结构。管理node、edge和全局state，确定workflow

Runnable：标准化的可执行组件。可以用于AI workflow中的特定任务，类似一个预制的StateGraph？

Messages：LangGraph中的消息类型（说明补充）

Human Message

from langchain_core.messages import HumanMessage

msg = HumanMessage(content="你好，请帮我计算 3 + 5")

代表用户输入的消息，是对话的起点。在LangGraph中，HumanMessage通常从输入端传入，或者通过interrupt机制在人工介入时产生。

AI Message

from langchain_core.messages import AIMessage

msg = AIMessage(content="3 + 5 的结果是 8")

代表AI/LLM返回的消息。AIMessage还可以携带tool_calls字段，表示LLM决定调用某个工具：

msg = AIMessage(
    content="",
    tool_calls=[{"name": "calculator", "args": {"a": 3, "b": 5}, "id": "call_001"}]
)

注意：当LLM想调用工具时，content通常为空字符串，工具调用信息放在tool_calls中。

System Message

from langchain_core.messages import SystemMessage

msg = SystemMessage(content="你是一个数学助手，请用简洁的方式回答问题。")

系统消息，用于设定AI的行为规则、角色定位或回答风格。通常放在对话的最开头，作为”幕后指令”。在LangGraph中，可以在构建prompt时动态注入SystemMessage。

Function Message（已废弃，建议用Tool Message替代）

from langchain_core.messages import FunctionMessage

msg = FunctionMessage(content="8", name="calculator")

这是旧版OpenAI function calling格式的返回值消息。当前LangChain/LangGraph推荐使用ToolMessage代替，FunctionMessage保留仅为向后兼容。

Tool Message

from langchain_core.messages import ToolMessage

msg = ToolMessage(content="8", tool_call_id="call_001", name="calculator")

工具执行后的返回结果。tool_call_id必须与AIMessage中tool_calls的id对应，这样LLM才能知道这个结果是哪个工具调用的返回值。这是当前LangGraph Agent中最常用的消息类型之一。

消息类型的整体使用流程：

from langchain_core.messages import HumanMessage, AIMessage, ToolMessage, SystemMessage

# 典型Agent一轮对话的消息流
messages = [
    SystemMessage(content="你是助手"),
    HumanMessage(content="3+5=?"),
    AIMessage(         # LLM决定调用工具
        content="",
        tool_calls=[{"name": "calc", "args": {"expr": "3+5"}, "id": "c1"}]
    ),
    ToolMessage(       # 工具执行结果
        content="8",
        tool_call_id="c1",
        name="calc"
    ),
    AIMessage(content="结果是8"),  # LLM根据工具结果生成最终回答
]

尝试编写图：Hello World Graph

最简单的图：两个节点，一条边，线性执行。

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 1. 定义State（图共享的数据结构）
class State(TypedDict):
    message: str

# 2. 定义节点函数（接收state，返回更新）
def node_a(state: State) -> dict:
    print("A: 收到消息 ->", state["message"])
    return {"message": state["message"] + " -> 经过A"}

def node_b(state: State) -> dict:
    print("B: 收到消息 ->", state["message"])
    return {"message": state["message"] + " -> 经过B"}

# 3. 构建图
builder = StateGraph(State)       # 用State类型初始化StateGraph
builder.add_node("A", node_a)     # 添加节点A
builder.add_node("B", node_b)     # 添加节点B
builder.add_edge(START, "A")      # START -> A
builder.add_edge("A", "B")        # A -> B
builder.add_edge("B", END)        # B -> END

# 4. 编译图
graph = builder.compile()

# 5. 运行图
result = graph.invoke({"message": "Hello"})
print("最终结果:", result)
# 输出:
# A: 收到消息 -> Hello
# B: 收到消息 -> Hello -> 经过A
# 最终结果: {"message": "Hello -> 经过A -> 经过B"}

要点：

  • State用TypedDict定义，保证类型安全；
  • 节点函数接收完整的state，返回一个dict表示对state的部分更新（LangGraph自动合并）；
  • START和END是LangGraph内置的虚拟节点，标记起点和终点；
  • 用invoke同步运行图，输入是一个符合State结构的dict。

构建图的基本操作

add_node —— 添加节点

builder.add_node("node_name", node_function)

将Python函数注册为图中的一个节点。节点名称（如"node_name"）是图中该节点的唯一标识。

add_edge —— 添加普通边

builder.add_edge("A", "B")   # A完成后，无条件执行B
builder.add_edge(START, "A") # 图启动时首先执行A
builder.add_edge("B", END)   # B完成后图结束

普通边表示无条件跳转：前一个节点完成后，立即执行下一个节点。

add_conditional_edges —— 添加条件边

def route(state: State) -> str:
    if state["score"] >= 60:
        return "pass"
    return "fail"

builder.add_conditional_edges(
    "A",           # 从节点A出发
    route,         # 路由函数：根据state决定下一个节点
    {
        "pass": "B",   # route返回"pass"时 -> B
        "fail": "C",   # route返回"fail"时 -> C
    }
)

条件边实现分支逻辑：根据当前state动态选择下一个节点。路由函数必须返回一个字符串，并且这个字符串必须在第三个参数的映射字典中存在。

compile —— 编译图

graph = builder.compile()

将StateGraph构建好的结构”固化”为一个可执行的图对象。编译时会检查图的结构合法性（如是否存在孤立节点、不可达节点等）。编译后还可以传入checkpointer参数（见下文）。

状态更新：Reducer与消息合并

默认情况下，节点返回的dict会覆盖state中同名的键：

class State(TypedDict):
    count: int

def node_x(state: State):
    return {"count": state["count"] + 1}  # 直接覆盖count

但对于messages列表，典型的期望是追加而非覆盖。这时需要用Annotated配合add_messages reducer：

from typing import Annotated
from langgraph.graph import add_messages
from langchain_core.messages import BaseMessage

class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    # add_messages 会将新消息追加到列表末尾，而不是替换整个列表

注意区分两种更新方式：

  • 普通字段（如int、str）：节点返回什么就覆盖什么；
  • Annotated + add_messages：新消息会追加到现有列表末尾（对ToolMessage还会自动匹配对应的AIMessage的tool_call_id）。

Checkpointer：状态持久化

图每次invoke之后，内存中的state就丢失了。要持久化状态，需要在compile时传入checkpointer：

from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()                 # 内存中持久化（测试用）
graph = builder.compile(checkpointer=checkpointer)

使用checkpointer后，每次运行需要提供config，其中包含thread_id：

config = {"configurable": {"thread_id": "user_001"}}

# 第一次对话
graph.invoke({"messages": [HumanMessage(content="我叫小明")]}, config)

# 第二次对话 —— 图会从上次的状态继续
graph.invoke({"messages": [HumanMessage(content="我叫什么名字？")]}, config)
# LLM能记住"小明"，因为messages列表通过checkpointer持久化了

要点：

  • thread_id用于区分不同的对话线程（不同用户或不同会话）；
  • MemorySaver仅存在内存中，重启即丢失；生产环境可用SqliteSaver或PostgresSaver；
  • 有了checkpointer，图就有了”记忆”——每次调用时state是之前所有历史消息的累积。
