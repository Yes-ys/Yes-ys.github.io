编译原理

大三的春季学期选修了编译原理这门课程，这篇blog用来记录其中关键的知识点

ch1：绪论

本课涵盖范围内容：

  1. 编译器概述
  2. 词法分析
  3. 上下文无关文法和语法分析
  4. 语法制导翻译和类型检查
  5. 运行时存储空间的组织和管理
  6. 中间代码生成

编译的概念：
将高级语言翻译成汇编语言或机器语言的过程；
1）高级语言：
类似数学定义或自然语言的简洁形式；
2）汇编语言：
在机器语言的基础上引入助记符；
3）机器语言：
可以被计算机直接理解的语言；
注意：
1）高级语言称为源语言；
2）汇编语言&机器语言称为目标语言；

语言处理系统：
处理过程：
源程序 by 预处理器 -> 经过预处理的源程序 by 编译器 -> 汇编语言程序 by 汇编器
-> 可重定位机器代码 by 链接器&加载器(载入库文件和其他可重定位目标文件) -> 目标机器代码
编译器的位置：
预处理器和汇编器之间

编译器的工作过程/结构：
两部分：
1）分析部分：分析源语言得到句子的语义；
2）综合部分：根据句子的语义生成目标语言；
注意：
1）分析部分也叫做前端，与源语言相关；
2）综合部分也叫做后端，与目标语言相关；
分析部分：
1）词法分析器：
接收字符流 输出记号流
主要任务：
1）扫描源程序字符，识别各个单词，确定单词的类型
2）将识别出的单词统一转换为词法记号（token）
3）token：<记号名，属性值>
常用的单词记号名分配方法：
1）关键字 1个单词1个记号名
2）标识符 多个单词1个记号名
eg：变量名x1 x2 x3，统一用id记号名，表示为<id,1> <id,2> <id,3>
3）常量 1种类型的多个单词用1个记号名
4）运算符 1种类型的多个单词用1个记号名 或 1个单词1个记号名
5）界限符 1个单词1个记号名
注意：
1）此处的单词是广义概念，显然从关键字～界限符都是单词
2）词法分析器会为我们构建出记号表（符号表）
注意：
1）构造符号表是编译器的一项重要工作；
2）符号表为每个标识符（token）保存一个记录的数据结构，允许快速根据记号名查找对应的数据（各种属性值）
3）属性值不是必须的，例如运算符的token通常就没有属性值 = -> <=>
eg：
position = initial + rate * 60 -> <id,1><=><id,2><+><id,3><*><60>
（这里<60>的完整写法是<int,60>，省略了记号名）
2）语法分析器：
接收记号流 输出语法树
主要任务：
1）从词法分析器输出的token序列中识别出各类短语，构造语法树；
注意：
1）语法树：描述句子的语法结构，内部节点表示运算，子节点表示运算对象；
2）表达式：任何一个表示符号&数都是表达式；
eg：
上个例子的token序列经过语法分析器之后输出的语法树；

3）语义分析器：
接收语法树 输出语法树
主要任务：
1）收集token的属性值；
2）类型检查；
eg：数组下标不是整数、操作符与操作数不匹配…
3）语义一致性检查；
eg：变量或过程未经声明就使用、变量或过程名重复使用…
4）隐式类型转换；
注意：
1）目前比较常用 语法制导翻译 将语法分析和语义分析有机组织起来
eg：
上个例子的语法树经过语义分析器处理后的状态；

4）中间代码生成器：
接收语法树 输出中间表示
中间代码的多种表示形式：
1）后缀表示；
2）图形表示；
3）三地址表示（每个指令的右部最多有三个操作数）；
中间代码必须具备的特点：
1）易于产生；
2）易于翻译成目标语言；
注意：
1）代码指令会确定运算次序；
2）在翻译成中间代码的时候可能需要添加临时变量保存结果；
3）三地址表示的中间代码不一定有三个运算对象；
eg：
上个例子语法树经过中间代码生成器之后生成的中间代码；

独立于机器的代码优化器：
接收中间表示 优化中间表示
综合部分：
1）代码生成器：
接收中间表示 输出目标机器代码
2）依赖于机器的优化器：
接收目标机器代码 优化目标机器代码
注意：
1）编译器的整体处理过程一共有七个阶段；
2）符号表会在多个阶段起作用；
3）按照前端后端划分的好处是有利于移植以及简化编译器的设计；
eg：
1）不同的前端工具生成完中间代码之后，可以复用同一套后端工具；
2）一套前端工具生成的中间代码，也可以用于多套后端工具；
4）编译器具体实现的时候，可以将几个阶段联合起来一起执行一遍；
eg：
前端读入一个输入，给出一个输出；
代码优化阶段可以反复执行很多遍；

翻译器、编译器和解释器：
翻译器：
能够将一种语言编写的程序翻译成语义等价的另外一种语言程序的软件系统；
编译器：
1）翻译器的一种；
2）目标语言通常比源语言低级；
（一种编程语言越接近机器语言，则该语言越低级）
解释器：
1）翻译器的一种；
2）不会生成目标程序，而是直接执行源程序所指定的运算；
（.py文件由python解释器直接执行，而不像.c文件要先生成.exe文件（本质机器语言）由操作系统执行）

ch2：词法分析

引入：
词法分析主要功能：
源程序字符流->词法记号流
手工构造词法分析器：
1）使用状态图转移图描述识别源程序中 单词<->词法记号 的结构；
2）将状态图翻译为程序语言；
涉及的理论：
正规式、状态转换图、自动机等
词法分析器与语法分析器的关系：
1）语法分析器是主导，词法分析器可以看作它的子程序；
2）两个模块是交错进行的，对于输入的源程序流，不断由词法分析器翻译为记号，形成记号流的同时语法分析器也在读取记号流进行语法分析；
词法分析器的任务：
1）将正确的单词识别为记号；
2）剥去注释和空白（制表符、回方程、空格等分隔符）；
3）将各阶段产生的错误信息与源程序联系起来；
4）宏的预处理可以在词法分析实现；
注意：
1）现代编译器通常有单独的预处理模块；
2）注释、空白和宏实际上可以在预处理模块处理（当然也可以由词法分析器处理）；

ch3：语法分析

语法分析器相关基本概念：
语法分析器的作用：将记号流构成语法结构；
注意：
1）记号流是由词法分析器转换原程序的字符流而来的；
2）语法结构是语法规则描述的合法的记号构成；
3）语法分析器简称分析器；

上下文无关文法：
引入：
词法分析器中描述词法规则时大多数使用正规式，但语法分析时很多语法规则不能用正规式描述，因此语法分析器使用上下文无关文法；
正规式的局限性：
1）正规式不能描述配对或者嵌套的结构；
eg：类似括号配对，前面的串受到后面的串出现情况的影响，这种情况正规式无法描述；
2）正规式只能使用终结符（尽管正规定义有非终结符，但是也必须对非终结符完全定义才能使用），不允许自嵌套；
基本定义：
上下文无关文法（CFG）定义为四元组$G = (V_T,V_N,S,P)$，其中：
1）$V_T$是终结符集合（i.e.，词法记号名）；
2）$V_N$是非终结符集合，与终结符集合交集为空；
3）$S$是开始符号，属于非终结符；
4）$P$是产生式集合；
产生式的形式：A->a，其中
1）左侧的A属于非终结符；
2）S至少在某产生式左侧出现一次；
3）a是终结符或非终结符；
注意：
1）上下文无关文法允许自嵌套，语言描述能力比正规式强；
2）所有的正规式都属于上下文无关语言，反之不成立；
3）上下文无关的含义：
句型中左部的非终结符可以在任何时候被替换，这种替换不依赖于句型中的其他符号（上下文）；
（每个产生式左部一定只有一个非终结符，只要找到符合产生式右边的串就可以将其规约成该非终结符）；
4）正规式可以描述的结构都可以用上下文无关文法描述；
为什么词法分析器要用正规式，不直接用上下文无关文法？
1）词法规则简单，正规式处理起来更加高效，相比用上下文无关文法效率更高；
2）站在整个源程序处理的角度来看，正规式描述关键字、标识符、常数等词法结构；而上下文无关文法可以用来描述配对、嵌套；
5）由于文法涉及的所有终结符和终结符都应该出现在产生式中（否则没有意义），因此在描述一个文法的时候通常可以直接简化为一系列产生式；
文法表示的约束：
终结符的表示：
1）字母表前面的小写字母a，b，c；
2）一些字符串，主要是id和表示一些关键字的字符串（例如while）；
3）数字1、2、3…；
4）标点符号和运算符号；
非终结符的表示：
1）字母表前面的大些字母A、B、C；
2）小写字母组成的字符串，例如expr、stmt；
注意：
一般的文法符号串使用希腊字母表示（类似数学中的未知数）；
文法的简化表述：
左侧非终结符相同的产生式可以写成一个，产生式右侧的文法串用“｜”连接；
推导：
推导符号：$\Rightarrow$
注意：一步推导、一步或多步推导、零步或多步推导的符号；
定义：用产生式作为重写规则，把符号串非终结符用产生式右侧的串来代替；
最左推导：每次都对当前串中最左侧的非终结符推导；
最右推导：每次都对当前串中最右侧的非终结符推导；
（最右推导又叫做规范推导）；
分析树：
定义：
1）推导的图形表示；
2）中间节点对应非终结符；
3）子节点由该终结符的这次推导所用产生式右部符号从左到右依次标记；
4）根节点由文法的开始符号标记；
注意：
1）又叫做语法分析树，但是要区分语法树；
一些概念与记号：
1）若文法为G，则文法产生的语言是L(G)；
2）上下文无关语言：上下文无关文法产生的语言；
3）句型：由某文法的开始符号推导出的串（可能含有非终结符）；
4）句子：只含有终结符的句型；
5）文法的等价：如果两个文法产生同样的语言，则两个文法等价；
文法递归：
定义：文法G中至少含有一个递归的非终结符时，G是递归文法；
1）$A \Rightarrow^* \beta A \gamma$，其中$\beta,\gamma$步同时为$\epsilon$，则A是递归的；
2）$A -> \beta A \gamma$，则A是直接递归的；
注意：
1）如果$\beta = \epsilon, \gamma \ne \epsilon$，则A对应是左递归/直接左递归的；
2）反之，A对应是右递归/直接右递归的；
（只有G是递归文法的时候，L(G)中的句子才是无穷的）；
文法二义性：
定义：该文法存在某个句子，有不止一个最左（最右）推导；
（对应两棵不同的分析树）；
注意：
1）文法是二义的对应的语言不一定是二义的，只有产生该语言的所有文法都是二义的，语言才是二义的；
2）有的时候允许二义文法的分析器，必须有消除二义性的规则；

编译原理中使用的文法都是上下文无关文法，因此下面全部将上下文无关文法简称文法；

文法的变换：
引入：
1）大部分语法分析器期望文法是无二义的，否则不能为句子选定一个唯一的语法分析树；
2）有的分析器（例如自上而下的语法分析器）还要求文法没有左递归和左公因子；
3）基于这些要求，我们要能够将二义的、有左递归、有左公因子的文法进行转换，得到等价的、满足要求的文法；
二义性问题：
定义：文法的一个句子对应了多个语法分析树；
解决思路：
1）从归约（句子->开始符）的角度来看，多个语法分析树是因为表达式中算符的优先级和结合性没有严格规定；
（例如1+2+3，先算1+2再算3+3和先算2+3再算1+5的结果是一样的，在过去的一些学科里这个顺序是不重要的，因为我们总能得到正确结果6；但是现在我们需要一套能处理所有类似范式的规范，因此应该严格规定结合性；例如规定+左结合，那么就只能先算1+2了；这样对应该句子的语法分析树就唯一了）
2）从推导（开始符->句子）的角度来看，我们应该限制每一步推导都只能有唯一的一个选择，这样从分析树根节点出发得到句子的整个分析树便只有一种模式，并且如此反向归约的时候便有了明确的优先级和结合性；
3）本质上是在解决优先级和结合性的问题；
解决方案：
1）解决优先级：对于文法中的每个“算符文法符号”引入新非终结符；
注意：
1）效果是增加一个子结构提高一级优先级，越靠近开始符号的文法符号优先级越低；
2）因此优先级低的算符所在的产生式应该写在前面；
2）解决结合性：使用递归产生式，递归的非终结符在产生式右侧表达式中出现的位置对应了结合性；
例如：
1）A->A+B，则+是左结合的；
2）A->B+A，则+是右结合的；
例子：
消除文法S->S and S | S or S | not S | true | false | (S)的二义性；
等价的非二义文法：
S -> S or E
E -> E and T
T -> not T | true | false | (S)
说明：
优先级 not > and > or
结合性 not右结合、and和or左结合
注意：
对于右侧符号串不含有“算符文法符号”的产生式，可以放在最后单独处理；
1）直接在最后一个产生式处理，如上例；
2）或在最后再附加一个产生式，统一处理，类似：
… T->not T | word
word -> true | false | (S)
注意：实际上上述消除文法二义性的解决方案主要是针对“表达式文法”，也就是针对 用于生成代表 表达式的语言的文法；下面还会介绍一种消除悬空else文法的二义性的解决方案；尽管不是完全一般的情况，但是这两种基本上就是我们会遇到的所有情况；
消除“悬空else文法”的二义性：
原始文法：
stmt -> if expr then stmt
| if expr then stmt else stmt
| other
二义性问题：
1）这种文法对于句型：if expr then if expr then stmt else stmt存在两个最左推导；
2）问题出现的本质原因是then的数量多于else，else在没有限制的情况下和两个then都可以配对；
解决思路：
1）规定在配对的then和else语句之间的语句必须是then、else配对的
解决方案：
1）stmt分成配对的，即MS和不配对的UMS，两种条件语句；
2）MS中then、else数量相等（包括0），UMS中then个数比else多；
3）利用MS、UMS实现解决思路中的效果：
stmt -> MS|UMS
MS -> if expr then MS else MS
-> other
UMS-> if expr then stmt
-> if expr then MS else UMS
（注意，other理解为0个then、else的非条件语句，例如源代码中if then中具体的语句）
左递归问题：
消除直接左递归：
直接左递归：文法存在形如 $A->A\alpha|\beta$ 的产生式；
解决方法：
将形如$A->A\alpha|\beta$ 的产生式改成：
$A -> \beta A’$
$A’-> \alpha A’ | \epsilon$
更一般的情况：
原始产生式 $A->A\alpha_1|A\alpha_2|…|A\alpha_m|\beta_1|\beta_2|…|\beta_n$
修改后：
$A ->\beta_1 A’|\beta_2 A’|…|\beta_n A’$
$A’->\alpha_1 A’|\alpha_2 A’|…|\alpha_m A’|\epsilon$
说明：本质上就是将左递归改为了右递归；
非直接左递归：
解决方法：
1）线将文法转换为等价的直接左递归文法；
1）非直接左递归一般的情况是：
1）有产生式c1 c2 .. cn；
2）其中c1的非终结符可以推导出含c2非终结符的串，c2的…，以此类推到cn的非终结符可以推导出含c1的非终结符串（且该非终结符在串的最左侧）；
3）将cn右侧中c1左侧的非终结符使用c1右侧的串代替，得到新产生式cn1；
（注意此时，仍然是非直接左递归，只是对应开始的是c2了c2…cn1，cn1右侧含有c2的非终结符）
4）将cn1右侧中c2左侧的非终结符使用c2右侧的串代替，得到cn2，依次类推直到得到cn(n-1)（将cn(n-2)右侧中c(n-1)左侧的非终结符用c(n-1)右侧的串替代，注意：这是对最后一个非直接左递归的处理，递归的非终结符相隔了一个产生式 c(n-1) cn(n-2)）
最终得到了等价的直接左递归；
一个具体的例子：
非直接左递归：
$S -> Aa|b$
$A -> Sd|\epsilon$
直接左递归：
$S -> Aa|b$
$A -> Aad|bd|\epsilon$
2）然后使用消除直接左递归文法的方法；
考虑无用符号的情况：
引入：
在处理左递归问题的时候，我们接触了两种文法的等价变换；
1）非直接左递归->直接左递归的变换；
2）直接左递归->不含左递归的变换；
实际上在这些变换的时候很可能产生无用符号，我们将这些无用符号删除，可以进一步简化文法的产生式；
无用符号的定义：
1）如果一个文法符号无法从开始符号推导出，则其为无用符号；
2）如果一个文法符号是非终结符，且从它继续推导永远不能推导出句子（仅含非终结符的句型），则其为无用符号；
（1很好理解，2是因为，如果无法终止的话则该文法符号一定不会出现在任何句子的推导中，因为只要是句子那么一定要终止，所以也是无用符号）
如何删除无用符号：
将含有无用符号（无论是左侧还是右侧）的产生式，直接删除；
例子：
文法：
S -> abS | abA | abB
A -> cd
B -> aB
C -> dc
其中：
1）C是无用符号，因为开始符号S无法推导至C；
2）B是无用符号，因为由B出发的推导不能终止；
删除后：
S -> abS | abA
A -> cd
有意在变换中构造无用符号，从而简化消除左递归后的文法：
说明：
这也是一种消除左递归的方法，但是过程中可以让原本文法中的一些文法符号变为无用符号，在消除左递归后可以删除他们，最终的文法会变得更加简洁；
操作方式：
具体省略，大致就是在非直接左递归->直接左递归那一步，将包含开始符号的产生式子作为cn，然后按部就班的处理，完成之后会发现cn(n-1)之前的产生式左侧的非终结符都变成无用符号了，因为从开始符号不能推导到它们，所以可以直接删除；
eg：
左因子问题：
左因子：
文法的左因子$\alpha$形如：
A -> $\alpha \beta_1 | \alpha \beta_2$
提左因子操作：
将上述含左因子的产生式子改成：
A -> $\alpha A’$
A’-> $\beta_1 | \beta_2$
为什么要提左因子：
1）含左因子的产生式，右侧有相同的模式$\alpha$，这就导致了分析器可能不清楚具体使用哪个来替换非终结符A；
2）提左因子的操作本质是利用引入了一个新非终结符的文法，让分析器延迟选择；（等到输入了足够多的信息之后再做选择）；
实际的例子：
原始文法：
stmt -> if expr then stmt|
| if expr then stmt else stmt|
| other
提左因子后的等价文法：
stmt -> if expr then stmt L | other
L -> else stmt | $\epsilon$

Chomsky 文法分类
分类方法：
1）0型文法；
说明：产生式左右的文法符号均可来源于 非终结符集&终结符集 并的闭包；
代表：短语文法；
2）1型文法；
代表：上下文有关文法；
说明：产生式左右的文法符号来源同0型 但是限制左侧文法符号的长度要小于右侧文法符号的长度；
3）2型文法；
代表：上下文无关文法；
说明：产生式左侧的文法符号只能来源于非终结符集 右侧的文法符号同0型
4）3型文法；
代表：正规文法；
说明：在2型的基础上进一步规定右侧文法符号非终结符与终结符的相对位置；
文法之间的关系：
0型包含1型包含2型包含3型；
（从上面的说明也可以看出来，限制是逐级加强的）；

自上而下分析：
分析方法：
1）从开始符号出发；
2）每次都选择当前文法串最左侧的非终结符进行替换；
3）替换用的产生式根据输入流的下一个终结符来确定；
注意：
1）这种方法反映在分析树上就是，自上而下，自左向右为输入串建立分析树；
2）本质上也是在为输入串，从开始符号出发寻找一个最左推导；
存在的问题：
1）不能处理左递归；（显然会陷入无限循环）；
2）需要很多临时变量来保存状态；
（因为这种方法也类似一个DFS，在一个无法扩展的、还未达成目标的状态，显然需要回溯，对应的就是很多临时变量，或者说内存空间）；
3）可能需要回溯，效率低；（回溯的原因是对于某输入符有多个可选产生式的文法，而选择可能错误）
LL(1)文法：
引入：
1）自伤而下分析的问题是在处理具体文法的时候出现的；
2）有左递归的文法让其陷入无限循环、对于某输入符有多个可选产生式的文法 让其可能出现错误选择从而回溯；
3）自然而然的 我们考虑有一类文法不会让这种分析方式出现这些问题；
4）这就是LL(1)文法；
LL(1)文法的定义：
结束符定义：
符号表示：$
符号含义：代表输入流的一系列符号的末尾，结束的位置；
注意：
1）前面提到了，分析的过程实际上就是从开始符号为输入串寻找一个最左推导；
2）因此实际上开始时$直接跟在开始符号S的后面；
3）$可以理解为一个特殊的终结符，它会隐式地出现在任何句子的末尾，标记句子的结束（尽管一般情况下我们不会注意到它，想一下C语言中字符数组的结尾标记’\n’）；
FIRST函数定义：
含义：
1）一个函数，用于把某个串映射到文法符号集合；
2）具体的函数$FIRST(\alpha) = {a|\alpha \Rightarrow^* a…,a\in V_T }$
注意：
1）FIRST函数用来针对文法符号寻找一个集合 其中的元素是该文法符号可以推导出的任意文法串的 开头的文法符号；
2）当然，如果可以推导出空串，那么$\epsilon$也会属于这个集合；
3）这个集合可以称为FIRST集合
确定FIRST集合的方法：
1）终结符的FIRST集合中只有它自己、以终结符开头的串的FIRST集合中只有其开头的终结符；
2）非终结符的情况按照FIRST集合定义；
3）注意零步或多步推导、终结符两个要求（也就是说如果开头是非终结符可能还要继续推下去）；
FOLLOW函数定义：
含义：
1）一个函数，用于把非终结符映射到文法符号集合；
2）具体的函数$FOLLOW(A) = {a|S$ \Rightarrow^* …Aa…,a\in V_T}$
注意：
1）FOLLOW函数是针对非终结符寻找一个集合 其中的元素是从开始符号推导出的任意串中 紧跟在非终结符后的终结符；
2）显然，FOLLOW(开始符号)中应该有$
3）这个集合可以称为FOLLOW集合；
4）显然epsilon不应该出现在任何非终结符的FOLLOW集合中，因为定义的句子中已经有了…Aa…的a，那么这个句子一定不会是epsilon（空句子）；
确定FOLLOW集合的方法：
1）开始符号的FOLLOW集合中有$；
2）如果有产生式 B -> alpha A beta，那么FIRST(beta)中除了epsilon以外的元素都在FOLLOW(A)中；
3）如果有产生式 B -> alpha A 或 “2）的情况+epsilon在FIRST(beta)中”（本质也是有B -> alpha A），那么FOLLOW(B)的一切元素都在FOLLOW(A)中；
注意：
1）FOLLOW集合可能要经过多轮计算才能得到完整的结果，从一个非终结符的FOLLOW集受到另一个非终结符的影响就可以看出；
2）显然2）、3）还是贴在原始的定义上的，透过定义可以更好的理解为什么要这样做；
思路：
1）自上而下分析存在两个问题：无限循环、回溯效率低；
2）避免这两个问题对文法的要求是：
1）没有左递归；
2）确保自上而下分析时针对输入流的当前文法符号只有一个能够选择的产生式（那么这个产生式一定是正确的产生式，不会有错误的产生式）；
3）基于这两个要求，利用FIRST集合和FOLLOW集合我们可以定义一种文法，适应至上而下分析，这就叫做LL(1)文法；
LL(1)文法定义：
对于一个不含左递归的产生式的文法，其任意两个产生式A->alpha | beta满足下列条件：
1）FIRST(alpha) 与 FIRST(beta)交集为空；
2）如果beta经过零步或多步推导能得到epsilon，那么FRIST(alpha)和FOLLOW(A)的交集为空；
说明：
1）上面的条件实际上都是为了让自上而下分析，处理当前输入流的文法符号a（终结符）时一定只有一个（正确的）选择（产生式）；
2）如果第一个条件不满足，假设a是两个FIRST交集中的一个终结符，仅看a，可以选A->alpha也可以选A->beta；
3）如果第二个条件不满足，也是如此A->alpha最终得到 a….，满足a在开头、A->beta最终得到 a beta … => a…，满足a在开头；
性质：
基于LL(1)文法的定义，它有一些性质；
1）不是左递归的；
2）不含左公因子（从第一个条件可以看出）；
3）不是二义的；
补充：
预测分析方法：
1）实际上在自上而下分析法一定不会回溯的时候，我们称它为预测分析法；
2）向前看1步：我们前面描述自上而下分析对于当前输入流的“下一个”终结符寻找产生式，本质上就是在自上而下DFS的时候向前看1步；
3）实际上类似的也有向前看k步的“预测分析法”，这时我们需要 构造类似 用于看1步的预测分析法的 LL(1)的 文法，称为LL(K)文法；
难以理解的一点：
本身自上而下分析，实际上是为了具体的文法的具体句子而服务的，我们需要识别这些句子；
但是我们又为了让自上而下分析成为“预测分析”而对文法添加限制，好像是文法在为预测分析服务；
实际上我们不用想的那么死，本质上自上而下分析的部分我们讨论的是 一类文法 它能在被自上而下分析处理时 使得处理方式表现出高效性；

自上而下分析的非递归实现
引入：
1）上一部分我们明确了如何构造一个可以由自上而下分析处理的文法；
2）并且简单了解了自上而下分析在处理这种文法的时候是怎么通过递归来做的；
3）这一部分我们改用非递归文法，并且更形式化、正式；
预测分析表结构：
列索引：所有可能的输入符号（终结符），注意包含$；
行索引：所有可能的非终结符；
表元素：非终结符的某个产生式；
非递归实现结构：
分别并行的维护：
1）栈：
1）存储处理的中间变量；
2）开始时栈中保存 $E（E是开始符号）；
2）输入：
1）存储剩余需要处理输入串；
2）开始时输入中保存 原始输入串$（注意以$结尾）；
3）输出：
1）存储上一时刻 输入串首字符 与 栈顶非终结符 在预测分析表中匹配对应的 产生式；
2）开始时为空（注意当前时刻的内容由上一时刻 输入与栈的内容确定）；
非递归实现的过程：
具体实现过程就是非递归实现结构动态增长，完善的过程：
1）初始化栈、输入、输出；
2）if 输出不为空: 将该时刻输出中 产生式 右侧的文法符号 从右到左 依次入栈（注意最左侧的在栈顶）；
3）
if 栈顶为终结符: 输入串首字符一定是对应的终结符（否则出错）识别该终结符 将其从输入串中移除
else 栈顶为非终结符: 根据当前输入、栈顶元素 根据预测分析表 确定下一时刻输出（无法确定则出错）；
4）重复2）直到栈顶只剩下 结束标志$；
构造预测分析表：
对文法中的所有产生式A -> alpha进行处理：
1）针对FIRST(alpha)中的每个终结符a 令M[A,a] = A -> alpha；
2）如果FIRST(alpha)中有epsilon 针对FOLLOW(A) 中的每个非终结符b 令M[A,b] = A -> alpha；
注意：
1）M[A,a]对应的是预测分析表元素；
2）FOLLOW(A)中的每个非终结符包括$；
3）在处理完所有产生式之后，预测分析表中可能存在没有定义的条目，对应为erro；
（意味着如果在预测分析的时候使用了这个条目就触发错误）；
一些错误情况的说明：
1）多重定义条目：
含义：显然根据上面构成预测分析表的方法，很可能会有多重定义的条目；
处理方法：
1）构造文法等价的LL(1)文法；
2）根据实际情况，删去多的条目，保留一条；（一般不会用）；
性质：预测分析表不存在多重定义条目 当且仅当 文法是LL(1)的；
2）空白条目；
错误恢复：
1）词法错误，如标识符、关键字或算符的拼写错误；遗漏字符串两端的引号；
2）语法错误，如算术表达式的括号不配对；
3）语义错误，如算符作用于类型不匹配的运算对象；
4）逻辑错误，如无穷的递归调用；C中“==”误写为“=”；
注意：
1）语法错误和少数语义错误（如类型不匹配）可在分析期间进行错误诊断和恢复；
2）但精确诊断语义错误和逻辑错误是困难的；

自下而上分析（移进-归约 分析）：
引入：
一个一般的归约过程：
1）根据一个初始时刻由终结符组成的文法串，选中其中的子串，选中一个产生式（该产生式的右侧即选中的该子串），将我们选中的子串转换为所选产生式的左侧文法串；
2）不断重复该过程，直到最终初始的文法串被完全转换为 初始符号；
归约过程的分类：
最左归约：
1）最右推导的逆过程；
2）实际上就是每次从左到右扫描子串，选中能够被归约的最左侧子串，进行归约；
注意：
1）最左归约也叫做规范归约；
2）自上而下分析实际上 通常使用的就是这个归约（LR分析，L自左向右 R最右推导的逆）
最右归约：
1）最左推导的逆过程；
2）实际上就是每次从右到左扫描子串，选中能够被归约到最右侧子串，进行归约；
自上而下分析的过程实际上就是利用这种归约的方法来不断处理子串，但是细化的时候要考虑如下三个问题：
1）什么时候进行归约；
2）归约哪一个子串；
3）归约成哪一个非终结符；
句柄：
定义：句柄是自上而下分析过程中，归约选中的子串；
（注意，自上而下分析实际上使用的是 “最左归约” 因此有下述的逆）
性质：
1）针对 某文法通过最右推导得到的句型 该句型的句柄一定是某产生式的右侧 且 将其转换为产生式左侧的文法串的过程 是最右推导逆过程的一步；
2）句柄的右边仅含有终结符；
3）如果文法二义，那么句柄可能不唯一；
3）如果文法不二义，那么句柄一定唯一；
说明：
1）自上而下分析方法也不是一种完全一般化的分析方式；
2）从 其使用“最左归约-最右推导的逆”作为分析时归约的基础 即可以看出 其针对性处理的文法串应该是：来自文法的最右推导的句型；
3）在明确处理的文法串之后，我们应该考虑：
1）如何找到右句型中将要归约的子串（可能有多个可以归约的子串）；
2）如果该子串是多个产生式的右侧文法串 应该使用哪一个产生式归约；
自上而下分析（移进-归约分析）的具体细节：
使用的数据结构：
并行的维护以下三个数据结构：
1）栈，保存文法符号；
2）输入缓冲区，待分析的输入文法串；
3）动作，两种：
1）将当前输入串首部的符号移入栈；
2）使用某个产生式对栈顶的句柄进行归约；
分析器的基本动作：
1）移进：把下一个输入符号移进栈；
2）归约：把栈顶的句柄归约为非终结符；
3）接受：宣告分析结果的成功；
4）报错：发现错误，调用错误处理例程；
可能存在的冲突：
1）移进-归约冲突：下一个输入符号是移进还是归约？
（发生在当前栈顶开始的文法串已经可以归约了，但是下一个输入符号移进后依然能够从栈顶开始构成能够归约的文法串的情况）
2）归约-归约冲突：有多个产生式，应该归约成哪一个；

LR分析器：
含义：
1）一种应用自下而上分析的分析器；
2）L：从左向右扫描；
3）R：构造最右推导的逆；
特点：
1）适用一大类上下文无关文法，且效率高；
2）分析栈中的文法符号总是形成一个活前缀；
3）栈中文法符号和剩余输入组成一个右句型；
（栈中文法符号就是该右句型的活前缀，栈顶一旦发现句柄就会被归约，所以栈中文法符号不会超过该句柄的右端）；
4）分析表的转移函数本质上是识别活前缀的DFA的转换函数；
5）是已知的最一般的无回溯的移进-归约方法（从LR文法的定义就可以看出，比较抽象…因此一般）；
6）能分析的文法类是预测分析法的真超集；
7）能及时发现语法错误；
8）手工构造分析表的工作量很大；
注意：
1）实际上是LR(k)分析器，k对应分析动作向前搜索的符号个数；
2）但是程序设计语言一般使用LR(1)分析器，省略符号表示k=1；
三种主要技术：
1）简单的LR方法（SLR）；
2）规范的LR方法；
3）向前看的LR方法（LALR）；
包含关系：
LR(1) > LALR(1) > SLR(1) > LR(0)；
LR分析表：
含义：类似自上而下分析的预测分析表，LR分析表也是用于分析过程中确定具体操作的分析表；
作用：告诉LR分析器，何时该移进、何时归约、归约时使用的产生式、何时出错、何时成功；
行索引：
状态，LR分析进行过程中会时刻维护一个辅助变量（状态）
列索引：
动作，针对所有终结符；
转移，针对所有非终结符；
表元素：
状态-动作（终结符）：
移进+状态，将输入缓冲区首字符压栈、将状态压栈；
归约+序号：
1）将栈顶状态、字符依次弹出；
2）使用序号对应的产生式归约字符，得到非终结符；
3）非终结符压栈，查表压入对应的状态；
状态-转移（非终结符）：
转移状态：非终结符压栈的时候 还需要压入的对应状态；
LR分析过程：
使用的数据结构：
1）和前面自下而上分析描述的相同，栈、输入缓冲区、动作；
2）注意栈中维护的总是字符+状态；
初始化：
1）栈：压入状态0；
2）输入缓冲区：存入输入的 终结符组成的 文法串 （注意末尾有$）；
3）动作：暂无；
分析过程：
1）取输入缓冲区首个字符 查分析表 确定动作：
2）
if 移进：压栈首字符 压栈移进状态；
if 归约：
将栈顶状态、字符依次弹出（弹出的数量依据所用产生式的右侧文法串确定）；
对字符使用归约序号对应的产生式，得到归约的非终结符；
非终结符压栈，查表（状态-转移），压入对应的状态；
（不会有转移的动作，转移的动作隐含在归约里）；
3）重复1、2这个过程，直到最终输入缓冲区中只剩下$，动作标志为接受，结束；
LR文法：
定义：能为之构造出所有条目都唯一的的LR分析表的文法；
活前缀：右句型的前缀，该前缀不超过右句型的最右端；
LR分析法和LL分析法对比：
1）文法限制：LR要求无二义，LL要求无二义、无左因子、无左递归；
2）分析表：LR是状态&文法符号（终结符、非终结符）表大，LL是非终结符&终结符 表小；
3）分析栈比较：LR栈中比LL多包含状态信息；

LR分析表构造：
引：
1）前面说明LR分析的时候需要使用LR分析表，这是关键所在；
2）LR分析表的构造有不同的方法，可以分为：
1）SLR分析；
2）规范LR分析；
3）LALR分析；
3）三种分析算法是一样的，但是构造分析表的方法不一样，分析能力也不一样；
（SLR分析能力最弱，但表构造最简单；）
项目：
全称：文法的LR(0)项目；
定义：在产生式的右侧某个地方加点的产生式；
说明：加点的目的是用来表示分析过程中的状态；
eg：
A -> XYZ对应有四个项目：
A -> ·XYZ
A -> X·YZ
A -> XY·Z
A -> XYZ·
注意：
特别的，A -> epsilon只有一个项目 A -> ·
SLR分析表构造：
SLR文法：能构造无冲突的SLR分析表的文法；
注意：
1）SLR文法都不是二义的；（有的非二义文法不能用SLR分析）；
2）SLR分析表中可能会出现移进-归约冲突、归约-归约冲突；
SLR分析器：使用SLR分析表的LR分析器；
前置概念：
拓广文法：
定义：在原始文法的产生式集合中新加入一个产生式子 S’ -> S，S是原文法的开始符号 对应的文法即原始文法的拓广文法；
作用：便于确定何时分析成功，当使用S’ -> S进行归约的时候就一定分析成功了；
项目集：项目的集合；
项目集族：一族项目的集合；
closure(I)：
1）I是一个项目集；
2）closure(I)是项目集的闭包，本身也是一个项目集，构造方式：
1）将I中的所有项目都加入集合；
2）如果集合中有项目 A -> alpha ·B beta，并且文法有产生式B -> gamma，那么将B -> ·gamma加入集合中（若还未加入），重复此过程，直到没有新项目加入该集合；
从文法构造识别活前缀的DFA：
说明：
1）DFA中的状态是项目集；
2）所有状态对应项目集，构成LR(0)项目集规范族；
3）项目集规范族中各个项目集的产生、转移是构造的关键，如下；
构造过程：
1）确定项目集I0 = closure(I)，其中I = {E’ -> ·E}；
2）选中还未处理完成的项目集，eg Ii；
注意：
1）处理完成的项目集指的是，其中的项目都有了3）-1）中的处理；
2）否则还未处理完成；
3）处理选中的项目集Ii，生成新状态Ij，并得到转移条件：
1）确定Ij中的核心项目，为Ii中的 ·在同一文法符号前的项目，将·向后移动一位后得到的项目；
2）使用核心项目求闭包，得到的所有项目组成的集合即Ij；
注意：
1）从Ii->Ij的转移条件即 “·向后移动一位”经过的文法符号；
（形式化的描述是Ij = goto(Ii,某文法符号)
2）Ij可能并非新状态，此时Ii 根据转移条件 转移到一个已有的状态；
4）不断重复2）、3）直到所有项目集都是“处理完成的”，于是就得到了一个DFA；
构造SLR分析表：
1）构造该文法的LR(0)项目集规范族（对应识别活前缀的DFA）；
2）对拓广文法编号，从S’->S开始为0，依次增加；
3）令项目集Ii对应分析表中的状态i，针对项目集中所有项目来确定 动作、转移：
状态i的动作Action确定方法：
1）if A -> alpha ·a beta and goto(Ii,a) = Ij，则ACTION[i,a] = sj；
（注意，ACTION表对应动作那一部分，行索引是状态，列索引是终结符）；
（注意，s代表移进，j代表紧接着压栈的状态）；
2）if A -> alpha· and A != S’ ，则对于所有FOLLOW(A)中的终结符a，有ACTION[i,a] = rx；
（注意，此处x是产生式 A->alpha的编号）；
3）特别的，if S’ -> S·在Ii中，则ACTION[i,$] = acc；
状态i都转移Goto确定方法：
1）if A -> alpha ·B beta and goto(Ii,B) = Ij，则GOTO[i,B] = j；
（注意，GOTO表对应转移那一部分，行索引是状态，列索引是非终结符）；
（注意，由于移进的时候也有“转移状态”，所以要区分开，但本质一样，j也是要压栈的状态）；
注意：
没有定义的条目都是erro

ch6：运行时存储空间的组织和管理

引入：
1）在编译的前端阶段（源语言->中间表示）主要的分析、操作取决于源语言的特性，和操作系统无关；
2）而后端阶段（中间表示->目标语言、程序），此时涉及到各种数据的分配（例如x86汇编语言会有一个代码片段专门用于声明分配的内存空间），用于存储全局变量、局部变量等等；
3）因此需要考虑这个过程中存储空间的组织和管理；
例——C编译器生成目标代码的内存分配策略；
从低内存->高内存分别是：
1）目标代码；
2）静态数据；
3）堆；
4）空闲内存；
5）栈；
（这部分对应全局存储分配策略）
注意：
1）栈底是从最高内存开始，向低内存增长的；
分别是：参数->返回值->控制链->访问链->机器状态->局部数据->临时数据；
（这部分对应局部存储分配策略）
2）除了栈以外，其他部分都是低内存向高内存增长的；
3）堆、栈是在运行过程中大小会改变的，目标代码和静态数据则不会，因此堆和栈在可用的空闲内存上此消彼长；

局部存储分配策略：
基本概念：在此基础上对存储分配进行讨论
过程：
函数与过程（这里的过程类似C语言中“{}”括起来的一个代码段）统一称为过程；
活动：
过程的活动，过程的一次执行称为过程的一次活动；
活动记录：
过程的活动需要可执行代码和存放所需信息的存储空间，称为活动记录；
注意：
1）实际上就是frame（帧）；
2）活动记录存放在栈上面，随着程序的运行不断改变；
3）不同语言的活动记录存储空间布局可能不同；
4）活动记录没有包含活动所需的全部信息，其中没有非局部变量、动态存储分配的变量（前者在静态数据区，后者在堆上）；
一个具体的例子：（栈上，高地址->低地址，7->1）
1） 临时数据：保存临时值，例如表达式中间结果；
2） 局部数据：作用域在本过程中的数据；
局部数据的分配：
1）按顺序连续分配；
2）使用偏移查找地址；
3）注意可能有不同的对齐要求；
例如定义结构体
有属性：char、long、char、double
对齐要求：char 1、long 4、double 8
则偏移分别是：
0（0～1）、4（4～7）、8（8～9）、16（16～24）
3） 机器状态：过程调用前的机器状态信息，如返回地址、需要恢复的寄存器内容等；
4） 访问链：过程嵌套语言的非局部数据的访问；
5） 控制链：指向调用者的活动记录；
6） 返回值：存放被调用过程返回给调用过程的值；
7） 参数：存放调用过程提供的实在参数，由被调用过程使用；
活动记录的长度：
1）活动记录对应了一个域，也是一个活动过程，在没有可变数组的情况下，其长度在编译时可以确定；
2）否则，调用时一定能确定活动记录的长度；
活动的生存周期：
从过程开始执行到结束的运行时间（注意，包含过程调用的子过程的执行时间）；
针对局部分配策略的概念：
过程定义：
声明，将过程体与过程名联系起来；
（例如函数、完整的程序）；
过程调用：
调用语句通过过程名执行被调用的过程体；
形式参数：
过程定义中使用的；
实在参数：
过程调用时替代形式参数的；
名字的作用域和绑定：
名字：
一个声明所声明符号名；
（例如int i; i就是声明的变量名）；
名字的作用域：
一个声明起作用的程序部分、程序过程；
注意：
1）即使在代码中一个名字只声明了一次，在执行的过程中，它可能会绑定到不同的数据对象（存储单元）；
（考虑递归的情况，很好理解）；
名字的绑定：
含义：
将声明的名字，e.g. i，映射到具体的存储单元；
说明：
1）这个过程是由“环境”（作用域、声明起作用的过程）来完成的；
2）在此基础上，从名字->具体值的完整过程是：
1）环境将名字映射到存储空间的具体存储单元（左值）；
（可以把左值理解成一个存储单元的地址）；
2）左值对应的存储单元的状态就是当前真实的值（右值）；
注意：
因此，赋值改变的是存储单元的状态，而不是环境，改变环境的是过程调用；
（绑定的）生命周期：
我们讨论名字有关的生命周期时，实际上讨论的是名字的一个绑定生效的时间；
（一个名字可以有多个绑定，决定当前索引到哪个存储单元，因此真正有意义的是名字的绑定）；
静态概念与动态概念对比：
静态-动态：
1）过程定义-过程活动；
2）名字声明-名字绑定；
3）声明的作用域-绑定的生命周期；
程序块：
定义：本身含有局部变量声明的语句（C中由{}包裹的代码段）；
注意：
1）程序块结构的声明作用域，是最接近的嵌套作用域规则给出；
2）程序块可以嵌套但是不能重叠；
3）并列的程序块不会同时活跃，变量可以重叠分配；
（使用栈空间，维护一个栈顶偏移量，一个程序块结束后偏移量回滚，让另一个程序块分配存储空间时覆盖于上一个程序块之上）；

ch7：中间代码生成

引入：
中间代码生成器的位置：是前端的最后一个模块；
接收：语法树；
输出：中间代码表示；
优点：使得编译器便于移植、优化、翻译；

常见的中间语言：
后缀表达式：
递归定义：
1）如果E是变量或者常量，则其后缀表达式是它本身；
2）如果E的形式为E1 op E2，则其后缀表达式为E1’ E2’ op，其中E1’、E2’是E1、E2的后缀表达式；
3）如果E的形式为(E)，则其后缀表达式为E；
优点：
1）不需要括号来改变优先级；
2）便于计算机处理（栈）；
图形表示：
语法树：儿子节点（非叶子）是运算符代表运算结果，中间节点（根节点）是运算符，叶子节点是变量、常量的运算对象；
性质：后序遍历语法树可以得到后缀表达式；
注意：
1）语法树是抽象语法树的简称；
2）分析树是语法分析树的简称；
有向无环图（DAG）：
在语法树的基础上，将相同的运算重复利用，在上层中间节点需要中间运算结果的时候，直接指向下层对应的运算；
（方向总是上层指向下层，所以在有向的角度来看是无环的）；
构造方法：
在语法树的语法执导翻译中，为mknode增加语义，在构造节点之前检查是否已经有相同的节点存在，如果是的话直接返回相同的节点，否则再构造；
三地址代码：
一般形式：x = y op z；
其中：
1）x、y、z是名字、常数或临时变量；
2）op是运算符；
注意：
1）三地址代码一次只处理一个运算符；
2）复杂表达式需要多个三地址代码组成的序列来处理；
3）中间结果保存在临时变量中；
eg：
x+yz的三地址代码：
1 t1 = yz
2 t2 = x+t1
优点：
1）使用临时变量使得三地址代码容易安排计算次序，后缀表达式的计算次序就不容易改变；
性质：
1）三地址代码实际上就是语法树的线性表示；
2）优化后的三地址代码实际上是dag的线性表示；（子节点的复用对应临时变量的复用）；
常用的三地址运算：
（）赋值语句 x = y op z， x = op y， x = y
（）无条件转移 goto L
（*）条件转移 if x relop y goto L
过程调用 param x 和 call p , n
过程返回 return y
索引赋值 x = y[i] 和 x[i] = y
地址和指针赋值 x = &y，x = *y 和 x = y
（标注为（）的是常用的）；
一个例子：
原始代码：
if (x < y)
z = x;
else
z = y;
z = z * z;
三地址：
if (x < y) goto L1 //这里改成 if (x >= y) goto L2，可以把下面的goto L2省略；
goto L2
L1: z = x
goto L3
L2: z = y
L3: z = z * z
（老师说goto是关键）
（要能够将各类语句翻译为三地址代码，可能涉及复杂的布尔表达式作为判断条件）
三地址指令：
三地址代码执行需要三地址指令，编译器用记录实现三地址指令，记录包括算符域和运算对象域，不同编译器具体实现不同；
静态单赋值形式（SSA）：
含义：一种便于某些代码优化的中间表示；
性质：
1）表示与三地址代码类似，有两个显著的区别；
2）SSA中所有赋值都是针对不同变量的赋值；
3）使用phi函数组合不同控制流来计算特定的值；
特点1eg：
原始：
p = a + b
q = p - c
p = q * d
p = e - p
q = p + q
SSA：
p1 = a + b
q1 = p1 - c
p2 = q1 * d
p3 = e – p2
q2 = p3 + q1
特点2eg：
原始：
if (flag) x = -1; else x = 1;
y = x * a;
SSA：
if (flag) x1 = -1; else x2 = 1;
x3 = phi(x1, x2);
y = x3 * a;

声明语句：
引入：
这里讨论关于中间表示的一个具体的翻译问题——如何翻译声明语句；
翻译过程：
1）为局部名字建立符号表条目；
2）为它分配存储单元；
3）符号表中会包含 名字的类型 被分配存储单元的相对地址信息；
（相对地址是针对基址的偏移）；

D S //D定义语句，S执行语句，按部就班

D -> id : T {enter ( id.name, T.type, offset); // 申明声明的变量了，要在符号表中记录，enter是先查找是否存在，不存在则向符号表添加
offset = offset + T.width } //offset是偏移量，根据变量的类型来确定
T -> integer {T.type = integer; T.width = 4 }
T -> real {T.type = real; T.width = 8 }
T -> array [ num ] of T1
{T.type = array (num.val, T1.type);
T.width = num.val times T1.width}
T -> uparrow T1 {T.type = pointer (T1.type); T.width = 4 }
（声明语句的翻译方案中，最重要的是offset怎么算）
说明：
1）emit是输出一条三地址代码字符串的含义；
